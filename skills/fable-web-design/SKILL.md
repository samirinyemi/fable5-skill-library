# Web Design

## Overview

A website is an argument: every viewport either advances the case or loses the reader. Fable's web method: **message architecture first, then a layout that dramatizes the message, then conversion mechanics that feel like hospitality, not pressure.**

**REQUIRED BACKGROUND:** fable-design-dna. For prestige-tier experiences, escalate to fable-award-winning-web.

## Message Architecture (before any layout)

1. Write the one-line promise a visitor must believe.
2. List the 3 objections that stop them believing it.
3. Order the page: Promise → Proof → Objection-handling → Ask. Every section maps to one of these or gets cut.
4. Write real headlines in the product's voice. Headlines are design decisions — a great layout cannot rescue "Unlock your potential."

## Layout Rules

- **First viewport = promise + one visual proof + one action.** Nothing else competes.
- Vary section rhythm: full-bleed → contained → asymmetric split → dense grid. Two identical section skeletons in a row is a flag.
- Break the card habit: proof can be a stat set in 120px type, a customer sentence set as editorial pull-quote, a product screenshot bleeding off-grid.
- Section spacing on a scale (96/128/160px desktop); breathing space between arguments, tight space within them.
- One CTA style repeated verbatim throughout — same words, same look. Competing CTAs split intent.

## Conversion Without Sleaze

- Specificity converts: "Deploy in 4 minutes" beats "Fast & easy".
- Social proof is placed at the moment of doubt (beside the CTA, after pricing), not dumped in a logo wall.
- Forms ask the minimum; every extra field is a tax on trust.
- The footer is a second chance, not a graveyard: restate the promise + ask.

## Quick Reference

| Zone | Job | Failure smell |
|---|---|---|
| Hero | State promise, earn 10 more seconds | Vague verbs, 2+ CTAs, carousel |
| Features | Prove the promise | Icon + title + blurb ×6 grid |
| Social proof | Kill doubt at the right moment | Logo wall with no context |
| Pricing | Make choosing easy | 5 columns, feature-matrix fatigue |
| Final CTA | Convert the convinced | New unrelated message |

## Common Mistakes

- Designing sections before writing the argument — produces beautiful pages that say nothing.
- Treating mobile as a stacked afterthought: 60%+ of traffic sees that version first. Design the narrow argument, too.
- Shipping the first hero. Write 5 headline options; the first is always the most generic.
